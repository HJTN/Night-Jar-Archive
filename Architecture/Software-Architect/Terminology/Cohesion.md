# 응집도 (Cohesion)

## Definition

하나의 모듈 내부 요소들이 얼마나 밀접하게 관련되어 있는가를 나타내는 척도.

응집도는 **높을수록 좋다**. 높은 응집도 = 단일 책임, 명확한 역할, 이해하기 쉬운 코드.

## Context / When to Use

모듈·클래스·컴포넌트 설계 시 "이 요소들이 한 단위로 묶여야 하는가?"를 판단할 때 기준이 되는 개념.

## 응집도 7단계 (낮은 순 → 높은 순)

| 단계 | 유형 | 설명 | 평가 |
|---|---|---|---|
| 1 | **우연적 (Coincidental)** | 관계 없는 요소가 우연히 한 모듈에 | 최악 |
| 2 | **논리적 (Logical)** | 비슷한 종류라는 이유로 묶임 (예: 모든 I/O 처리) | 나쁨 |
| 3 | **시간적 (Temporal)** | 같은 시점에 실행된다는 이유로 묶임 (예: 초기화 로직) | 나쁨 |
| 4 | **절차적 (Procedural)** | 순서가 있는 작업들이 묶임 | 보통 |
| 5 | **통신적 (Communicational)** | 같은 데이터를 다루는 요소들이 묶임 | 보통 |
| 6 | **순차적 (Sequential)** | 한 요소의 출력이 다음 요소의 입력이 되는 경우 | 좋음 |
| 7 | **기능적 (Functional)** | 단 하나의 잘 정의된 기능만 수행 | 최선 |

## 높은 응집도의 이점

- **이해 용이**: 하나의 책임만 가지므로 코드 파악이 빠름
- **테스트 용이**: 격리된 기능 단위로 단위 테스트 작성 쉬움
- **재사용성 향상**: 명확한 목적의 모듈은 다른 컨텍스트에서도 재사용 가능
- **변경 영향 최소화**: 변경 시 영향 범위가 해당 모듈로 한정됨
