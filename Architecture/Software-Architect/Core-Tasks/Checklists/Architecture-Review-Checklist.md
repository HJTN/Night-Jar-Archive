# Architecture Review Checklist

> 아키텍처 리뷰 시 [[Quality-Attributes|품질 속성]]별로 확인해야 할 항목

---

## 1. 기능 요구사항 충족

- [ ] 모든 비즈니스 기능 요구사항이 설계에 반영되었는가?
- [ ] 데이터 흐름이 비즈니스 프로세스와 일치하는가?
- [ ] 외부 시스템·API 통합이 모두 정의되었는가?

---

## 2. 성능 (Performance)

- [ ] 예상 트래픽(TPS, DAU)에서 목표 응답시간을 달성할 수 있는가?
- [ ] 병목 가능 지점이 식별되었는가?
- [ ] 캐싱 전략(Redis, CDN 등)이 적절히 정의되었는가?
- [ ] 데이터베이스 인덱스·쿼리 최적화 계획이 있는가?

---

## 3. 가용성 & 복원력 (Availability & Resilience)

- [ ] 단일 장애 지점(SPOF)이 제거되었는가?
- [ ] 장애 시 복원 전략(Circuit Breaker, Retry, Fallback)이 정의되었는가?
- [ ] SLO 목표를 달성 가능한 설계인가?
- [ ] 재해 복구(DR) 전략 및 RTO/RPO가 정의되었는가?

---

## 4. 확장성 (Scalability)

- [ ] 트래픽 증가 시 수평 확장(Scale-out)이 가능한가?
- [ ] 상태(State)가 적절히 분리되어 있는가? (Stateless 컴포넌트)
- [ ] 데이터베이스가 읽기·쓰기 부하를 처리할 수 있는가? (Read Replica 등)
- [ ] 비동기 처리·메시지 큐로 부하를 분산하는 구조가 있는가?

---

## 5. 보안 (Security)

- [ ] 인증(Authentication) 및 인가(Authorization) 메커니즘이 정의되었는가?
- [ ] 민감 데이터(PII, 결제 정보) 암호화 계획이 있는가?
- [ ] API 노출 범위가 최소화되어 있는가? (최소 권한 원칙)
- [ ] OWASP Top 10 리스크가 설계 단계에서 검토되었는가?
- [ ] 네트워크 경계(VPC, 방화벽, TLS) 정책이 정의되었는가?

---

## 6. 유지보수성 (Maintainability)

- [ ] 컴포넌트 간 [[Coupling|결합도]]가 낮은가?
- [ ] 각 컴포넌트의 책임이 명확히 분리되어 있는가? (높은 [[Cohesion|응집도]])
- [ ] 기술 스택이 팀 역량·기존 표준과 일치하는가?
- [ ] 배포 전략(블루-그린, 카나리, 롤링)이 정의되었는가?

---

## 7. 관찰 가능성 (Observability)

- [ ] 로깅 전략이 정의되었는가? (구조화 로그, 로그 레벨)
- [ ] 메트릭 수집 및 대시보드 계획이 있는가?
- [ ] 분산 트레이싱(Distributed Tracing) 계획이 있는가?
- [ ] 알림(Alert) 조건이 정의되었는가?

---

## 8. 비용 (Cost)

- [ ] 인프라 비용(클라우드 리소스)이 추산되었는가?
- [ ] 예상 트래픽 증가 시 비용 증가 추세가 허용 범위인가?
- [ ] 불필요한 리소스 낭비 지점이 없는가?

---

## 9. 의사결정 기록

- [ ] 주요 기술 결정이 [[ADR]]로 기록되었는가?
- [ ] 채택하지 않은 대안과 그 이유가 문서화되었는가?
- [ ] 미결 사항(Open Questions)이 명시적으로 표기되었는가?

---

## Notes

- 모든 항목이 Yes일 필요는 없음. 체크 결과를 근거로 **우선순위 논의**가 목적
- 리뷰 결과는 [[Confluence]]에 기록하고 Must Fix 항목은 [[Jira]]에 등록
- [[Architecture-Review-Process]] 절차에 따라 리뷰 진행
