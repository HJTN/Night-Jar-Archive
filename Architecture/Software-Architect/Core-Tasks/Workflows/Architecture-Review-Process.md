# Architecture Review Process

> 설계 산출물을 이해관계자와 검토하고 피드백을 반영하는 흐름

## Overview

아키텍처 리뷰는 설계가 요구사항·품질 기준·팀 역량을 충족하는지 구현 전에 검증하는 프로세스다. 문제를 설계 단계에서 발견하는 비용이 구현 후보다 10배 이상 저렴하다. 주기적(분기·릴리즈 전)으로 실시하거나, 중요한 설계 변경 시 임시로 진행한다.

## Steps

1. **리뷰 준비** (2~3일 전)
   - 아키텍처 설계 문서 및 다이어그램 배포
   - 리뷰어 선정: 시니어 아키텍트, 시니어 개발자, 인프라 담당, 보안 담당
   - 리뷰 목적 명시: 신규 설계 검토 / 변경사항 검토 / 정기 리뷰

2. **컨텍스트 공유** (10~15분)
   - 비즈니스 배경 및 목표 설명
   - NFR 및 아키텍처 드라이버 공유
   - 주요 제약 조건(일정·비용·팀 역량) 공유

3. **아키텍처 워크스루** (30~45분)
   - C4 다이어그램 기반 설명: Context → Container → Component 순서
   - 주요 데이터 흐름 설명
   - 핵심 ADR(기술 결정) 설명

4. **질문 & 피드백 수렴** (30분)
   - 설계 결정의 근거 질문
   - 대안적 접근 제시
   - 리스크·누락 사항 식별

5. **[[Architecture-Review-Checklist]] 항목 검토**
   - 성능·가용성·보안·확장성·유지보수성 항목 체계적으로 확인
   - 체크 결과를 리뷰 문서에 기록

6. **액션 아이템 도출 및 분류**

   | 분류 | 의미 | 처리 방식 |
   |---|---|---|
   | **Must Fix** | 구현 전 반드시 해결 | 다음 리뷰 전 수정 |
   | **Should Fix** | 중요하지만 협의 가능 | 일정 내 처리 |
   | **Nice to Have** | 개선 제안 | 백로그 등록 |

7. **결과 문서화**
   - 리뷰 일시, 참석자, 주요 피드백, 액션 아이템을 [[Confluence]]에 기록
   - Must Fix 항목은 [[Jira]] 이슈로 등록

8. **피드백 반영 & 후속 확인**
   - 수정된 설계로 Must Fix 해결 확인
   - 필요 시 소규모 후속 리뷰 진행

## 리뷰 유형

| 유형 | 시점 | 범위 |
|---|---|---|
| **신규 시스템 리뷰** | 구현 시작 전 | 전체 아키텍처 |
| **변경사항 리뷰** | 중요 변경 발생 시 | 변경 영향 범위 |
| **정기 아키텍처 리뷰** | 분기·릴리즈 전 | 전체 건강도 점검 |
| **긴급 리뷰** | 장애 후 또는 리스크 발견 시 | 문제 영역 집중 |

## Inputs

- 아키텍처 설계 문서 및 C4 다이어그램
- NFR 목록 및 아키텍처 드라이버
- ADR 목록 (주요 결정 기록)

## Outputs

- 리뷰 결과 문서 (피드백, 결정 사항, 액션 아이템)
- 수정된 아키텍처 문서
- 등록된 [[Jira]] 이슈 (Must Fix 항목)

## Notes

- 리뷰는 비판의 자리가 아닌 협업의 자리 — 설계자를 공격하지 않고 설계를 개선한다
- 리뷰어는 사전에 문서를 읽고 참석하는 것이 기본 예의
- "왜 이 결정을 했는가?"를 ADR에서 먼저 확인하면 리뷰 효율이 높아짐
- 소규모 팀: [[Design-Decision-Checklist]]로 대면 리뷰를 보완할 수 있음
