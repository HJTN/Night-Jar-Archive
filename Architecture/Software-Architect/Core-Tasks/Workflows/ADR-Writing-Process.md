# ADR Writing Process

> 아키텍처 결정 사항을 [[ADR]] 포맷으로 문서화하는 절차

## Overview

ADR(Architecture Decision Record)은 중요한 기술 결정과 그 배경·대안·결과를 기록하는 경량 문서다. 팀이 "왜 이 결정을 했는가"를 미래에도 이해할 수 있게 하고, 동일한 논쟁의 반복을 막는다. 되돌리기 어려운(One-Way Door) 결정에 집중해 작성한다.

## Steps

1. **결정 필요 인식** — 중요한 기술 결정 상황 식별
   - 되돌리기 어려운가? → ADR 작성
   - 팀이 반복적으로 같은 질문을 하는가? → ADR 작성
   - 새 팀원이 이해하기 어려운 구조가 있는가? → ADR 작성

2. **컨텍스트 작성 (Context)**
   - 이 결정이 필요한 배경, 비즈니스 요구사항, 기술적 제약 기술
   - "지금 우리가 처한 상황은 무엇인가?"

3. **대안 탐색 (Options)**
   - 가능한 대안 3가지 이상 탐색
   - 각 대안의 장단점을 표로 정리

4. **의사결정 (Decision)**
   - 선택한 방향을 한 문장으로 명확히 서술
   - "우리는 X를 선택한다."

5. **근거 작성 (Rationale)**
   - 선택 이유를 구체적으로 기술
   - 채택하지 않은 대안과 그 이유 명시

6. **결과 작성 (Consequences)**
   - 이 결정으로 발생하는 긍정적 영향
   - 발생하는 부정적 영향 또는 새로운 제약
   - 다음에 해결해야 할 과제

7. **팀 리뷰 & 승인**
   - 아키텍트·시니어 개발자에게 리뷰 요청
   - 이의 없으면 상태 → `Accepted`

8. **문서 저장**
   - Git repo 내 `docs/adr/ADR-{번호:0001}.md` 형태로 저장 권장
   - 또는 [[Confluence]] ADR Space에 저장

## ADR 번호 및 상태 관리

```
ADR-0001: Accepted         ← 현재 유효한 결정
ADR-0002: Accepted
ADR-0003: Superseded by ADR-0007  ← 번복된 결정
ADR-0007: Accepted         ← ADR-0003을 대체
```

## Inputs

- 비즈니스 요구사항 및 기술 제약
- 팀 토론 내용·회의록
- PoC 결과 또는 기술 조사 보고서

## Outputs

- ADR 문서 (번호·상태·컨텍스트·결정·근거·결과 포함)
- [[Confluence]] 또는 Git repo에 저장된 ADR 이력

## Notes

- ADR은 작성 후 내용을 수정하지 않는다. 번복 시 새 ADR을 작성하고 이전 ADR을 Superseded로 표기
- 모든 결정이 ADR 대상은 아님 — 되돌리기 어렵거나 팀 전체에 영향을 주는 결정에 집중
- Git에 저장 시 코드와 결정 이력이 함께 관리되어 PR과 연결 가능
- [[Design-Decision-Checklist]]로 대안 탐색 단계를 보강할 수 있음
