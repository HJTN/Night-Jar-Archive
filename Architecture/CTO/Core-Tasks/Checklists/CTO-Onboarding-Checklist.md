# CTO Onboarding Checklist

> 신임 CTO가 새 조직에 합류한 후 첫 30·60·90일에 수행해야 할 항목

## 개요

새 CTO가 효과적으로 온보딩하기 위한 체크리스트. **"빠른 변화"보다 "깊은 이해"가 우선**이라는 것이 핵심 원칙이다. 첫 90일은 신뢰를 쌓고 맥락을 파악하는 시간이다.

---

## ✅ 첫 30일 — 경청하고 이해하기

### 사람 & 조직

- [ ] 직속 리포트(엔지니어링 리더들)와 1:1 미팅 완료
- [ ] 각 팀 리더에게 "잘 되고 있는 것 / 개선해야 할 것 / 당신이라면 가장 먼저 바꿀 것"을 물었는가?
- [ ] CEO·CPO·CFO 등 경영진과 개별 미팅 완료 (기술에 대한 기대 파악)
- [ ] 현재 팀 구성, 역량 분포, 공석 파악 완료

### 기술 현황

- [ ] 주요 시스템 아키텍처 파악 (다이어그램 또는 직접 인터뷰)
- [ ] [[Technical-Debt]] 현황 파악 (시스템별 주요 부채 항목)
- [ ] 최근 6개월 주요 장애 및 Post-Mortem 리뷰
- [ ] 현재 [[Tech-Radar]] 또는 기술 스택 목록 파악
- [ ] CI/CD 파이프라인, 배포 프로세스, 모니터링([[Datadog]]/[[Grafana]]) 현황 확인

### 비즈니스 정렬

- [ ] 회사의 현재 비즈니스 목표와 전략 문서 검토
- [ ] 기술 조직이 비즈니스에 어떻게 기여하는지 이해
- [ ] 현재 [[OKR]] 또는 목표 체계 파악

### 프로세스 & 문화

- [ ] 스프린트/칸반 프로세스, 코드 리뷰 문화 직접 관찰
- [ ] [[Engineering-Culture]] 현황 체감 (심리적 안전감, 투명성 수준)
- [ ] 현재 사용 중인 도구 목록 파악

---

## ✅ 첫 60일 — 진단하고 계획 세우기

### 진단

- [ ] 팀 전체 eNPS 첫 측정
- [ ] 채용 파이프라인 및 공석 우선순위 파악 → [[Engineering-Hiring-Process]]
- [ ] 인프라·클라우드 비용 현황 분석 → [[FinOps]] 관점 진단
- [ ] 보안 취약점 및 컴플라이언스 갭 파악
- [ ] [[DORA-Metrics]] 현재 기준값(Baseline) 측정

### 계획

- [ ] 단기(90일) 기술 우선순위 초안 작성
- [ ] 빠른 성과(Quick Win) 3가지 식별 및 실행 시작
- [ ] 채용이 필요한 포지션 우선순위 결정
- [ ] 경영진과 단기 계획 공유 및 피드백 수렴
- [ ] [[Tech-Radar]] 갱신 또는 신규 작성 착수

### 관계

- [ ] 주요 외부 파트너·벤더 미팅 완료
- [ ] 고객팀(CS, Sales)과 미팅 — 기술 관점 고객 이슈 파악
- [ ] 직속 리포트와 1:1 주기·형식 확립

---

## ✅ 첫 90일 — 방향 제시하고 실행하기

### 전략 수립

- [ ] 기술 비전 초안 작성 및 팀 공유 → [[Tech-Vision-Template]] 활용
- [ ] 분기/연간 [[OKR]] 초안 작성 및 경영진 정렬
- [ ] [[Technology-Strategy-Planning]] 프로세스 착수 또는 기존 프로세스 개선
- [ ] [[Tech-Radar]] 첫 버전 발행 또는 갱신

### 조직 & 문화

- [ ] 조직 구조 개선이 필요하다면 방향 제시 (섣부른 재편 금지)
- [ ] Blameless Postmortem 또는 정기 기술 공유 세션 신설 검토 → [[Engineering-Culture]]
- [ ] 온보딩 가이드 현행화

### 성과 측정

- [ ] 30·60일 대비 팀 분위기·진행 상황 비교
- [ ] CEO·경영진에게 90일 회고 발표 (배운 점 + 앞으로의 방향)

---

## 주의사항

> "첫 90일에 큰 변화를 일으키고 싶은 충동을 참아라. 먼저 신뢰를 쌓아라."

- **결정을 서두르지 말 것** — 충분히 이해하기 전의 변화는 저항을 만든다
- **경청이 최우선** — 질문하고 답을 들어라. 아직 답을 줄 단계가 아니다
- **Quick Win 선택 중요** — 팀이 함께 성공 경험을 쌓을 수 있는 것을 고른다
- **이전 조직 방식 강요 금지** — 현 조직의 맥락을 이해하고 적용 여부를 판단한다
