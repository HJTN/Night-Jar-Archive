# Tech Evaluation Checklist

> 새 기술·도구·플랫폼 도입 시 타당성·리스크·비용을 점검하는 항목 ([[Build-vs-Buy]] 의사결정과 연계)

## 1. 필요성 검토

- [ ] 현재 해결하려는 문제가 명확히 정의되어 있는가?
- [ ] 기존 도구·라이브러리로 해결 가능한지 먼저 검토했는가?
- [ ] 도입하지 않았을 때의 비용(기회비용, 유지비용)을 산정했는가?

## 2. 기술 성숙도

- [ ] 프로덕션 레퍼런스가 충분한가? (유사 규모 회사의 사용 사례)
- [ ] 오픈소스라면 GitHub Stars·커밋 빈도·이슈 해결 속도 확인했는가?
- [ ] 버전 안정성: Major 버전이 안정화되었는가? (v0.x는 주의)
- [ ] 벤더가 상용이라면 재무 안정성·지원 정책을 확인했는가?

## 3. 팀 역량 및 학습 비용

- [ ] 현재 팀에 해당 기술 경험자가 있는가?
- [ ] 온보딩 학습 비용(시간)을 추산했는가?
- [ ] 공식 문서·튜토리얼·커뮤니티 질이 충분한가?

## 4. 보안 및 컴플라이언스

- [ ] 알려진 CVE(취약점)가 없는가? 패치 주기가 적절한가?
- [ ] 데이터 처리 방식이 내부 보안 정책에 부합하는가?
- [ ] GDPR·개인정보보호법 등 컴플라이언스 요건을 충족하는가?
- [ ] 오픈소스 라이선스가 상업적 사용에 문제 없는가? (GPL 여부 확인)

## 5. 운영 및 유지보수

- [ ] 모니터링·알림·로깅 지원이 되는가?
- [ ] 장애 시 디버깅 방법이 명확한가?
- [ ] 업그레이드 경로가 명확하고 하위 호환성이 유지되는가?
- [ ] 벤더 종속(Vendor Lock-in) 리스크를 파악했는가?

## 6. 비용

- [ ] 라이선스·구독 비용을 산정했는가? (1년·3년 TCO)
- [ ] 인프라 추가 비용(클라우드 리소스 등)을 포함했는가?
- [ ] 팀 교육 비용과 초기 구축 비용을 포함했는가?

## 7. 통합성

- [ ] 기존 기술 스택과 통합이 가능한가?
- [ ] [[Tech-Radar]]에서 해당 기술의 위치를 확인했는가?
- [ ] 마이그레이션이 필요할 경우 탈출 전략(Exit Strategy)이 있는가?

## 8. 최종 결정

- [ ] [[Build-vs-Buy]] 관점에서 대안을 비교했는가?
- [ ] 파일럿(PoC) 범위와 성공 기준을 정의했는가?
- [ ] 이해관계자 승인을 받았는가?

## Notes

- 모든 항목이 Yes일 필요는 없음. 리스크를 인지한 상태에서 의사결정하는 것이 목표
- 평가 결과는 [[Confluence]]에 기록해 의사결정 근거로 보관
- [[Tech-Radar]]에 해당 기술 추가 여부 결정 후 [[Tech-Radar-Review-Process]] 연계
