# PoC Execution Process

> PoC 목표 정의·환경 구성·실험·결과 평가·보고 절차

## Overview

[[PoC]](Proof of Concept)는 기술적 불확실성을 줄이기 위한 소규모 실험이다. "이 기술이 우리 상황에서 작동하는가?"를 확인하는 것이 목적이며, 완성도 높은 코드나 기능 구현이 목적이 아니다.

PoC 결과는 아키텍처 결정(ADR)의 근거 자료가 된다.

## Steps

1. **PoC 필요성 식별**
   - 어떤 기술적 불확실성을 해소하려는가? 구체적으로 정의
   - PoC 없이 결정할 수 없는가? → 그렇다면 PoC 진행
   - PoC 범위와 기간을 제한 (일반적으로 1~2주)

2. **PoC 목표 & 성공 기준 정의**
   - 가설(Hypothesis): "X 기술을 사용하면 Y 요건을 충족할 수 있다"
   - 성공 기준(Success Criteria): 수치로 표현 (예: "초당 1,000건 처리 가능")
   - 실패 기준도 명확히: 이 결과가 나오면 대안 탐색

3. **PoC 범위 및 설계**
   - 최소한의 실험으로 최대 불확실성 해소
   - 성공 기준을 측정할 수 있는 테스트 시나리오 작성
   - 실험 환경 명세 (규모, 데이터 볼륨, 시나리오)

4. **환경 구성**
   - PoC 전용 환경 구성 (프로덕션과 분리)
   - 최소한의 인프라 구성 (비용 최소화)
   - 필요한 도구·라이선스 확보

5. **실험 실행**
   - 성공 기준을 측정하는 테스트 실행
   - 예상치 못한 제약이나 문제 발견 시 즉시 기록
   - 데이터 수집: 성능 지표, 오류율, 설정 복잡도

6. **결과 분석**
   - 성공 기준 달성 여부 평가
   - 예상과 다른 결과 분석
   - 새로 발견된 제약·리스크 정리

7. **대안 비교** (필요 시)
   - 다른 후보 기술도 동일한 기준으로 실험
   - 결과를 비교표로 정리

8. **PoC 보고서 작성**
   - 가설, 실험 방법, 결과, 결론 문서화
   - 아키텍처 결정 권고안 포함
   - → [[PoC-Proposal-Template]] 참고

9. **결정 & 다음 단계**
   - PoC 결과를 바탕으로 아키텍처 결정
   - ADR로 결정 사항 문서화 (왜 이 기술을 선택/탈락했는가)
   - 프로덕션 적용을 위한 작업 계획 수립

## Inputs

- 기술적 불확실성 목록
- [[NFR]] 및 성공 기준
- 후보 기술·솔루션 목록

## Outputs

- PoC 보고서 (가설·방법·결과·결론·권고안)
- 비교 분석표
- ADR 초안 (기술 결정 근거)

## Notes

- PoC 코드는 버릴 각오로 작성 — 품질보다 속도
- 범위 크리프 주의: 처음에 정의한 성공 기준에만 집중
- 실패한 PoC도 가치 있다 — "이 기술은 X 이유로 우리 상황에 맞지 않음"을 확인했다
- 시간이 정해진 타임박스(Time-box) 방식으로 진행
