# Solution Design Process

> 고객 요구사항 분석부터 솔루션 아키텍처 제안서 완성까지의 흐름

## Overview

솔루션 아키텍트는 고객(내부 사업부 또는 외부 고객)의 비즈니스 문제를 이해하고, 가용한 기술과 제품을 조합해 최적의 솔루션을 설계·제안한다. 결과물은 솔루션 아키텍처 문서(SAD)와 제안서(Proposal) 형태로 나온다.

## Steps

1. **킥오프 & 요구사항 수집**
   - 고객과 킥오프 미팅: 비즈니스 목표, 현재 문제, 기대 결과 확인
   - 기능 요구사항(FR) 목록 작성
   - [[NFR|비기능 요구사항(NFR)]] 수집: 성능·가용성·보안·확장성 목표
   - 제약 조건 파악: 예산, 일정, 기존 기술 스택, 규정 준수

2. **현황 분석 (As-Is)**
   - 고객의 현재 시스템 구조 파악
   - 현재 아키텍처의 문제점·병목·기술 부채 식별
   - 기존 인프라 환경 파악 (온프레미스/클라우드)

3. **솔루션 범위 정의**
   - 해결할 문제와 해결하지 않을 문제 명확히 구분 (In Scope / Out of Scope)
   - 마일스톤 및 단계적 접근 여부 결정

4. **아키텍처 대안 탐색**
   - [[Reference-Architecture]] 참조
   - 최소 2~3개 대안 탐색
   - 각 대안의 [[Trade-off|트레이드오프]] 분석: 비용, 복잡도, 리스크, 마이그레이션 난이도

5. **아키텍처 설계 (To-Be)**
   - 선택한 방향으로 솔루션 설계
   - [[High-Availability]] 및 DR 전략 정의
   - [[Scalability|확장성]] 전략 정의
   - 보안 아키텍처 정의 (인증·인가·암호화·네트워크)

6. **다이어그램 작성**
   - 시스템 컨텍스트 다이어그램 (C4 Level 1)
   - 솔루션 아키텍처 다이어그램 ([[Draw-io]], [[Lucidchart]])
   - 네트워크·인프라 다이어그램

7. **비용 추산**
   - 구현 비용 (개발·설계·컨설팅)
   - 인프라 운영 비용 (TCO: Total Cost of Ownership)
   - 클라우드 비용 계산기 활용 (AWS Pricing Calculator 등)

8. **리스크 평가**
   - 기술적 리스크, 운영 리스크, 일정 리스크 식별
   - 리스크별 경감 방안 수립

9. **[[PoC]] 계획** (필요 시)
   - 기술적 불확실성이 높은 부분을 PoC로 검증 계획 수립

10. **솔루션 문서화**
    - SAD(Solution Architecture Document) 작성 → [[Solution-Architecture-Document-Template]]
    - 경영진용 요약 (Executive Summary) 작성
    - [[Confluence]]에 게시

11. **솔루션 리뷰 & 발표**
    - [[Solution-Review-Checklist]]로 자체 검토
    - 고객·이해관계자에게 발표 (Presentation)
    - 피드백 반영 및 최종 확정

## Inputs

- 고객 비즈니스 요구사항 (BRD 또는 RFI)
- [[NFR]] 목록 (성능·가용성·보안 목표)
- 기존 시스템 현황 문서

## Outputs

- SAD (Solution Architecture Document)
- 아키텍처 다이어그램 세트
- 비용 추산서 (TCO)
- 리스크 목록 및 경감 방안
- [[PoC]] 계획 (해당 시)

## Notes

- "좋은 아키텍처"가 아니라 "고객 제약 내 최선의 아키텍처"를 목표로
- 비용은 이르게, 자주 논의 — 마지막에 비용 이야기가 나오면 설계를 다시 해야 할 수 있음
- [[Trade-off]]를 숨기지 말고 고객이 인지한 상태에서 결정하도록
