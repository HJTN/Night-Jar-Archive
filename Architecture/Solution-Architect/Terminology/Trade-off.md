# 트레이드오프 (Trade-off)

## Definition

하나의 이점을 얻기 위해 다른 것을 희생하는 설계상의 타협.

아키텍처는 "완벽한 설계"가 없다 — 모든 선택은 무언가를 얻고 무언가를 포기한다. 솔루션 아키텍트의 핵심 역량은 트레이드오프를 명확히 이해하고, **고객이 인지한 상태에서 최선의 선택을 하도록 돕는 것**이다.

## Context / When to Use

모든 아키텍처 결정 시. 특히 [[NFR]] 간 충돌이 발생할 때, 대안 탐색 단계에서 비교 기준으로 활용.

## 대표적인 트레이드오프 사례

| 선택 | 얻는 것 | 포기하는 것 |
|---|---|---|
| MSA | 독립 배포, 확장성 | 운영 복잡도, 분산 시스템 비용 |
| 높은 가용성 (다중화) | 장애 내성, SLO 충족 | 비용 증가, 설계 복잡도 |
| 캐시 레이어 추가 | 성능 향상 | 데이터 일관성 (Eventual Consistency) |
| 이벤트 기반 아키텍처 | 낮은 결합도, 비동기 처리 | 디버깅 복잡, 최종 일관성 |
| 자체 구축 | 맞춤 기능, 벤더 독립 | 높은 개발·운영 비용 |
| SaaS 도입 | 빠른 구현, 낮은 운영 부담 | 벤더 종속, 기능 제한 |

## CAP 정리

분산 시스템의 근본적 트레이드오프:

```
Consistency (일관성)
     ↙         ↘
Availability  Partition Tolerance
(가용성)      (네트워크 분리 허용)
```

**셋 중 2개만 동시에 달성 가능**. 네트워크 분리(P)는 현실에서 피할 수 없으므로, 실제 선택은 CP 또는 AP:

- **CP** (일관성 + 분리 허용): MongoDB, HBase — 일관성 우선, 일부 요청 거부 가능
- **AP** (가용성 + 분리 허용): Cassandra, DynamoDB — 가용성 우선, 최종 일관성

## 트레이드오프 분석 방법

| 단계 | 내용 |
|---|---|
| 1. 후보 대안 나열 | 최소 2~3개 대안 |
| 2. 평가 기준 정의 | NFR 기반 (성능·비용·복잡도·유지보수성) |
| 3. 비교표 작성 | 각 대안을 기준별로 비교 |
| 4. 우선순위 반영 | 고객의 최우선 NFR에 가중치 부여 |
| 5. 결정 & 문서화 | 선택 이유와 포기한 것을 명시 |
