# PoC (Proof of Concept, 개념 증명)

## Definition

기술적 실현 가능성을 검증하기 위한 소규모 시험 구현. "이 기술이 우리 상황에서 실제로 작동하는가?"를 확인하는 것이 목적이며, 완성도 높은 코드나 기능 구현이 아니다.

## Context / When to Use

- 검토 중인 기술·아키텍처의 불확실성이 높을 때
- 아키텍처 결정에 앞서 기술적 근거가 필요할 때
- [[RFP]] 응답 전 제안 솔루션의 실현 가능성을 내부 검증할 때

## PoC vs MVP vs Pilot

| 구분 | 목적 | 완성도 |
|---|---|---|
| **PoC** | 기술 실현 가능성 검증 | 매우 낮음 (버릴 각오) |
| **MVP** | 최소 기능으로 시장 반응 검증 | 낮지만 출시 가능 수준 |
| **Pilot** | 소규모 실사용 환경에서 전체 솔루션 검증 | 높음 |

## PoC 성공 기준 정의

좋은 PoC는 **수치로 표현된 성공 기준**이 있다:

- "초당 1,000건의 메시지를 Kafka로 처리할 수 있는가?"
- "GraphQL 서버가 기존 REST API 대비 응답시간 20% 개선되는가?"
- "외부 ML 모델 API를 기존 파이프라인에 통합할 수 있는가?"

## PoC 실행 원칙

- **타임박스**: 1~2주 제한. 기간 내 결론을 내림
- **최소 범위**: 성공 기준만 테스트하는 최소한의 구현
- **실패도 가치**: "이 기술은 X 이유로 우리 상황에 맞지 않음" = 성공적인 PoC
- **문서화 필수**: 결과를 ADR 또는 PoC 보고서로 기록
