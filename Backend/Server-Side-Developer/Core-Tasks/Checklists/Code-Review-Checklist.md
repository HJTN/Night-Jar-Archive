# Code Review Checklist

> 서버 코드 리뷰 시 확인해야 할 항목

## 비즈니스 로직 정확성

- [ ] 요구사항 티켓의 모든 케이스가 구현되어 있는가?
- [ ] 엣지 케이스(빈 입력, 경계값, 동시 요청)가 처리되어 있는가?
- [ ] 기존 기능에 의도치 않은 영향을 주지 않는가?
- [ ] 트랜잭션 경계가 올바른가? (너무 넓거나 좁지 않은가?)
- [ ] 롤백이 필요한 상황에서 @Transactional이 적용되어 있는가?

## DB 쿼리 및 데이터 접근

- [ ] N+1 문제가 없는가? (연관 엔티티 로딩 방식 확인)
- [ ] 조회 조건에 해당하는 인덱스가 존재하는가? (신규 쿼리의 경우)
- [ ] 불필요하게 모든 컬럼을 조회(SELECT *)하지 않는가?
- [ ] 대량 데이터 처리 시 페이징이 적용되어 있는가?
- [ ] DB 마이그레이션 파일이 포함되어 있고, 롤백 가능한가?

## 예외 처리

- [ ] 예외가 적절히 처리되거나 상위로 전파되는가?
- [ ] 예외 catch 후 빈 블록이 없는가? (삼키는 예외 없음)
- [ ] 사용자에게 노출되는 에러 메시지가 내부 구현 정보를 포함하지 않는가?
- [ ] 외부 API 호출, DB 조회 등 실패 가능한 작업에 적절한 예외 처리가 있는가?

## 보안

- [ ] 인증(Authentication)이 필요한 API에 인증이 적용되어 있는가?
- [ ] 인가(Authorization)가 올바른가? (다른 사용자의 리소스에 접근 불가)
- [ ] SQL Injection 취약점이 없는가? (파라미터 바인딩 사용 확인)
- [ ] 입력값 검증이 있는가? (`@Valid`, `@NotNull`, 길이 제한 등)
- [ ] 민감 정보(비밀번호, 토큰)가 로그에 출력되지 않는가?
- [ ] 민감 정보가 응답 JSON에 포함되지 않는가?

## 성능

- [ ] 루프 내에서 반복적인 DB 조회가 없는가?
- [ ] 자주 호출되고 변경이 드문 데이터에 캐싱이 고려되었는가?
- [ ] 불필요한 객체 생성이나 비효율적인 자료구조 사용이 없는가?
- [ ] 대용량 데이터를 메모리에 한 번에 올리지 않는가?

## 테스트

- [ ] 핵심 비즈니스 로직에 대한 단위 테스트가 있는가?
- [ ] 예외 케이스 테스트가 포함되어 있는가?
- [ ] 테스트가 외부 의존성 없이 빠르게 실행되는가?
- [ ] 테스트 커버리지가 의미 있는 케이스를 다루는가? (커버리지 수치만 채우는 형식적 테스트 없음)

## 코드 가독성 및 네이밍

- [ ] 메서드/변수명이 의도를 명확히 표현하는가?
- [ ] 하나의 메서드가 하나의 책임만 가지는가?
- [ ] 매직 넘버/매직 스트링 대신 상수 또는 Enum을 사용했는가?
- [ ] 주석이 "무엇"이 아닌 "왜"를 설명하는가?
- [ ] 중복 코드가 없는가?

## Notes

- 리뷰어는 "이렇게 바꿔야 해"보다 "이런 이유로 이 방향을 고려해보세요"처럼 제안형으로 피드백한다
- 모든 코멘트가 동일한 비중이 아니다. 필수(Blocking)/권장(Non-blocking)/참고(Nit) 구분을 명시하면 작성자가 우선순위를 파악하기 쉽다
- 좋은 코드에도 칭찬 코멘트를 남긴다
