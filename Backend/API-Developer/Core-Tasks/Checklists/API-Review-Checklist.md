# API Review Checklist

> API 설계 및 구현 리뷰 시 확인해야 할 항목

## Checklist

### URI 설계
- [ ] 리소스명은 명사·복수형을 사용했는가 (`/users`, `/orders`)
- [ ] URI에 동사가 없는가 (`/getUsers` 금지)
- [ ] 계층 관계가 2단계 이하로 표현됐는가 (`/users/{id}/orders` 까지)
- [ ] 소문자와 하이픈(-) 사용, 언더스코어(_) 미사용
- [ ] 버전이 URI에 명시됐는가 (`/v1/users`)

### HTTP 메서드
- [ ] GET은 조회만, 서버 상태를 변경하지 않는가
- [ ] POST는 새 리소스 생성에 사용했는가
- [ ] PUT은 리소스 전체 교체, PATCH는 부분 수정으로 올바르게 사용했는가
- [ ] DELETE는 멱등하게 설계됐는가 (두 번 호출해도 404 또는 204 반환)

### 응답 코드
- [ ] 생성 성공은 201, 응답 본문 없는 성공은 204를 사용했는가
- [ ] 존재하지 않는 리소스는 404를 반환하는가
- [ ] 인증 실패(401)와 권한 없음(403)을 구분했는가
- [ ] 입력 오류는 400, 비즈니스 규칙 위반은 422를 사용했는가
- [ ] Rate Limit 초과 시 429와 `Retry-After` 헤더를 반환하는가

### 인증 / 보안
- [ ] 보호가 필요한 엔드포인트에 인증(JWT/API Key)이 적용됐는가
- [ ] 권한이 필요한 작업에 인가(Role 체크)가 적용됐는가
- [ ] 민감 정보(password, SSN, 카드번호)가 응답에 노출되지 않는가
- [ ] [[Rate-Limiting]]이 설정됐는가

### 요청 구조
- [ ] 필수 파라미터와 선택 파라미터가 명확히 구분됐는가
- [ ] 입력값 유효성 검증이 서버에서 이루어지는가 (클라이언트 검증만으로 부족)
- [ ] Path variable과 Query parameter 사용이 적절한가
  - Path: 리소스 식별 (`/users/{id}`)
  - Query: 필터링·정렬·페이지네이션 (`?status=active&page=1`)

### 응답 구조
- [ ] 응답 필드명이 camelCase로 일관됐는가
- [ ] 날짜/시간은 ISO 8601 형식인가 (`2026-06-28T10:30:00Z`)
- [ ] 목록 응답에 [[Pagination]] 메타데이터가 포함됐는가
- [ ] null 필드 처리 방식이 일관됐는가 (제외 또는 null 명시)

### 에러 응답
- [ ] 에러 응답 구조가 전체 API에서 통일됐는가
  ```json
  { "error": { "code": "USER_NOT_FOUND", "message": "..." } }
  ```
- [ ] 에러 메시지가 문제 해결에 도움이 되는가 (스택 트레이스 노출 금지)
- [ ] 에러 코드(영문 상수)가 정의됐는가

### 페이지네이션
- [ ] 대량 데이터 반환 API에 페이지네이션이 적용됐는가
- [ ] 페이지 크기에 상한선이 있는가 (예: 최대 100건)
- [ ] 무한 스크롤 API는 [[Pagination|Cursor 방식]]을 사용했는가

### 문서화
- [ ] OpenAPI 스펙에 모든 엔드포인트가 문서화됐는가
- [ ] 각 파라미터에 설명과 예시 값이 포함됐는가
- [ ] [[Swagger]] UI에서 직접 테스트 가능한가

## Notes

- 체크리스트는 구현 완료 후 PR 제출 전 셀프 체크 용도로 활용
- 팀에서 합의된 API 규칙이 있다면 이 체크리스트에 추가
- [[API-Release-Checklist]]는 배포 직전 최종 점검용으로 별도 사용
